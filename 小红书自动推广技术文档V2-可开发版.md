# 小红书自动推广技术文档 V2（可开发版）

## 1. 文档目标

本文档在 V1 的基础上，补全可直接进入开发阶段的关键设计，覆盖以下内容：

- 数据表设计
- 抓取任务流设计
- 内容生成 Prompt 模板
- 万相出图规则
- 小红书自动发布的浏览器自动化方案

本文档默认第一版目标如下：

- 每天自动发布 `10-20` 篇小红书图文笔记
- 内容方向为 `Ozon 热点解读 + 榜单分析 + 选品建议 + 运营干货`
- 内容来源以外部公开数据源为主
- 视觉生成优先使用万相
- 发布方式采用浏览器自动化登录并模拟发布

## 2. 技术选型建议

### 2.1 后端与任务系统

推荐技术栈：

- 后端：`Laravel` 或 `FastAPI`
- 定时任务：`cron + queue worker`
- 队列：`Redis`
- 数据库：`MySQL`
- 文件存储：本地磁盘或对象存储
- 浏览器自动化：`Playwright`
- 内容生成模型：便宜中文 LLM
- 图片生成模型：万相文生图

如果你后续希望和 `umlink` 现有系统集成，建议优先使用 `Laravel + MySQL + Redis + Playwright 独立服务` 的组合。

原因：

- 方便与现有 PHP 系统协同
- 任务调度、后台管理、队列消费容易接入
- Playwright 可单独作为 Node 服务运行，降低主系统耦合

### 2.2 服务拆分建议

第一版建议拆成 4 个逻辑服务，不一定要求物理拆分：

- `collector`：负责抓取外部热点
- `content-engine`：负责清洗、选题、生成文案
- `image-engine`：负责调用万相生成图片
- `publisher`：负责小红书自动发布

第一版如果追求上线速度，也可以先做成单体应用，内部按模块划分。

## 3. 系统模块与职责

### 3.1 外部热点采集模块

职责：

- 抓取 Ozon 公开页面、榜单页、搜索建议、行业资讯、中文平台公开标题
- 将原始数据统一写入 `source_raw_items`
- 对外部数据打上来源、时间、主题标签

产出：

- 可清洗的原始热点数据

### 3.2 热点清洗与聚类模块

职责：

- 去重
- 关键词归一
- 同义词合并
- 相似主题聚类
- 低质量内容过滤

产出：

- 结构化热点主题包

### 3.3 选题编排模块

职责：

- 按栏目和时段生成当天的发布计划
- 控制选题重复率
- 控制栏目比例
- 计算内容优先级

产出：

- `topic_candidates`
- `publish_plan_items`

### 3.4 文案生成模块

职责：

- 基于主题包和栏目模板生成标题、正文、标签、导流文案、封面短句
- 对文案做轻量改写，降低 AI 感和重复率

产出：

- `content_drafts`

### 3.5 图片生成模块

职责：

- 生成封面图
- 生成辅助信息图
- 下载图片并保存到素材目录

产出：

- `asset_images`

### 3.6 质检模块

职责：

- 审核内容安全性
- 审核营销风险
- 检查重复度
- 检查导流强度
- 评分是否允许发布

产出：

- `qa_results`

### 3.7 发布模块

职责：

- 使用 Playwright 自动打开小红书创作页
- 上传图片
- 填写标题、正文、标签
- 执行发布
- 记录截图和结果

产出：

- `publish_records`

## 5. 抓取任务流设计

### 5.1 任务总流程

```text
定时触发采集
-> 抓取页面/接口
-> 解析内容
-> 原始数据入库
-> 清洗去重
-> 聚类成主题
-> 生成选题
-> 进入发布计划
```

### 5.2 任务拆分建议

建议将任务拆成以下几个 Job：

1. `FetchSourceJob`
- 按数据源配置抓取页面

2. `ParseSourceJob`
- 将原始页面转成结构化字段

3. `NormalizeTrendJob`
- 统一关键词、标题、类目命名

4. `ClusterTrendTopicsJob`
- 聚合相似主题

5. `BuildTopicCandidatesJob`
- 基于主题生成可发选题

6. `BuildDailyPublishPlanJob`
- 生成当天发布计划

7. `GenerateContentDraftJob`
- 生成文案

8. `GenerateImageAssetJob`
- 调用万相出图

9. `RunQaCheckJob`
- 质检和评分

10. `PublishXiaohongshuJob`
- 执行浏览器自动发布

### 5.3 调度建议

建议调度频率：

- 抓取任务：每 `30-60` 分钟一次
- 聚类与选题：每 `2-3` 小时一次
- 发布计划生成：每天凌晨一次
- 内容生成：按发布计划提前 `2-6` 小时生成
- 自动发布：按时段消费发布队列

### 5.4 抓取优先级

P0：

- Ozon 搜索建议页
- Ozon 榜单页
- Ozon 类目页
- 行业资讯站标题列表

P1：

- 中文平台公开标题
- 竞品博客
- Telegram / YouTube 公开讨论

P2：

- 趋势工具
- 更多外文社区

### 5.5 抓取去重规则

建议使用三层去重：

- URL 去重
- 标题归一后哈希去重
- 正文摘要相似度去重

建议阈值：

- 标题相似度大于 `0.85` 视为重复候选
- 主题关键词交集比例大于 `0.7` 视为同主题

## 6. 选题与文案生成规则

### 6.1 栏目类型

第一版固定 4 类：

- `hot_keyword`
- `ranking_analysis`
- `product_selection`
- `operation_tips`

### 6.2 每天内容配比

建议：

- 热搜词解读：6 篇
- 榜单拆解：5 篇
- 选品建议：5 篇
- 运营技巧：2-4 篇

### 6.3 标题规则

要求：

- 长度控制在 `16-22` 字优先
- 不使用过度夸张语气
- 不使用强导购表达
- 必须含一个核心信息点
- 尽量具有“结论感”或“提醒感”

标题示例方向：

- 最近 Ozon 这类词明显在升温
- 这份新品榜里有几个信号要注意
- 这个类目看着热，其实不适合盲冲
- 做 Ozon 的标题，先别急着堆关键词

### 6.4 正文结构规则

正文建议采用 5 段结构：

1. 开头一句结论
2. 数据或现象描述
3. 为什么会这样
4. 对卖家的具体建议
5. 软性导流收尾

正文要求：

- 总字数建议 `250-450`
- 每段不要过长
- 避免口号式空话
- 必须能让读者快速看到一个明确观点

## 7. Prompt 模板

以下 Prompt 为第一版可直接落地模板，建议统一维护版本号。

### 7.1 选题生成 Prompt

用途：

- 将热点主题转成多个小红书选题

模板：

```text
你是一名熟悉 Ozon 跨境运营的小红书内容策划。

请根据下面的热点主题，生成 5 个适合小红书图文笔记的选题。

要求：
1. 面向人群是 Ozon 卖家、跨境运营人员。
2. 内容方向偏干货、分析、提醒，不要写成广告。
3. 每个选题都要具体，不要空泛。
4. 选题要能自然导流到运营工具，但不要直接营销。
5. 返回 JSON 数组，每项包含：
   - topic_title
   - topic_angle
   - topic_brief
   - column_type

热点主题：
{{trend_summary}}
```

### 7.2 标题与正文生成 Prompt

用途：

- 生成完整笔记文案

模板：

```text
你是一名专业的小红书内容写手，熟悉 Ozon 平台和跨境卖家运营场景。

请基于给定选题生成一篇适合小红书图文笔记发布的内容。

要求：
1. 风格偏专业、克制、真实，避免营销腔。
2. 标题长度控制在 16-22 字优先。
3. 正文总字数控制在 250-450 字。
4. 正文必须包含：
   - 开头一句结论
   - 2-4 个具体观察点
   - 1 段针对卖家的建议
   - 结尾的软性导流
5. 导流要自然，只能表达“如果平时要更高效跟踪、整理、上架、分析，可以借助工具处理”，不能直接硬卖。
6. 输出 JSON，字段如下：
   - title
   - cover_text
   - body_text
   - hashtags
   - cta_text

选题：
标题：{{topic_title}}
角度：{{topic_angle}}
摘要：{{topic_brief}}
栏目：{{column_type}}

参考热点信息：
{{source_context}}
```

### 7.3 去 AI 味改写 Prompt

用途：

- 对草稿做二次去模板化

模板：

```text
请将下面这篇小红书文案改写得更自然一些。

要求：
1. 保留原有观点，不改变事实方向。
2. 减少模板感、总结腔和过于整齐的表达。
3. 不要加入夸张词，不要加入鸡汤式口号。
4. 让它更像有经验的运营人在分享观察。
5. 输出纯文本正文。

原文：
{{body_text}}
```

### 7.4 质检 Prompt

用途：

- 对文案做发布前审核

模板：

```text
你是小红书内容审核助手，请对以下文案做风险与质量判断。

请返回 JSON，字段如下：
- risk_score: 0-100
- repeat_score: 0-100
- ai_tone_score: 0-100
- publish_score: 0-100
- blocked: true/false
- block_reason: string
- suggestions: string array

判断标准：
1. 是否存在明显广告导流。
2. 是否存在绝对化、夸大表达。
3. 是否过度像 AI 生成。
4. 是否适合小红书图文语境。

标题：
{{title}}

正文：
{{body_text}}
```

## 8. 万相出图规则

### 8.1 第一版图片策略

第一版不追求复杂设计，优先保证：

- 风格统一
- 成本可控
- 出图稳定
- 能支撑高频发布

建议图片类型：

- 每篇至少 1 张封面图
- 可选 1 张辅助信息图

### 8.2 出图比例建议

建议优先使用：

- 封面：`3:4`
- 备用：`1:1`

原因：

- 更适合小红书封面视觉习惯
- 容易做标题留白

### 8.3 栏目视觉风格规则

`热搜词解读`

- 视觉关键词：趋势、放大、上升、关键词卡片
- 颜色方向：浅背景 + 高对比重点色

`榜单拆解`

- 视觉关键词：榜单、排序、数据感、信息卡
- 颜色方向：简洁、理性、偏数据产品风格

`选品建议`

- 视觉关键词：商品场景、类目感、判断提示
- 颜色方向：偏实拍感，不要强海报感

`运营技巧`

- 视觉关键词：工作台、分析、方法、流程
- 颜色方向：更克制，偏工具感

### 8.4 Prompt 结构

建议统一结构：

```text
主体场景 + 信息风格 + 构图要求 + 光线/氛围 + 禁止项
```

封面图 Prompt 模板：

```text
为“小红书 Ozon 干货图文封面”生成一张高质量竖版配图。
主题：{{topic_title}}
风格要求：简洁、专业、真实、带信息感，不要夸张海报风，不要过强营销感。
画面主体：{{visual_subject}}
构图：适合竖版封面，主体明确，留出标题区域，视觉重心稳定。
质感：干净、清晰、现代，适合跨境电商运营内容。
```

负向 Prompt 建议：

```text
低清晰度，文字模糊，手部畸形，人物脸部异常，过度饱和，海报感太强，夸张广告风，杂乱背景，强 AI 感，水印，logo，密集小字
```

### 8.5 万相调用规则

建议：

- 模型优先选稳定版
- `n=1`
- 封面图先只生成 1 张，失败再重试
- 测试阶段开启 `prompt_extend`
- 控制统一尺寸，避免后处理复杂化

建议参数：

```text
size: 1040*1472 或同等 3:4 竖版
n: 1
watermark: false
prompt_extend: true
```

### 8.6 图片后处理建议

第一版推荐加一个轻量后处理步骤：

- 统一压缩
- 统一重命名
- 可选加固定封面文案层

如果后面要把封面做得更稳，建议改成：

- AI 生成背景图
- 本地程序叠加统一标题样式

这样比让模型直接生成带完整文字的封面更稳定。

## 9. 小红书自动发布的浏览器自动化方案

### 9.1 技术选型

推荐：

- `Playwright`
- 持久化浏览器用户目录
- 单独 Node 发布服务

原因：

- 对现代前端站点兼容更好
- 处理上传、等待、选择器更稳定
- 比 Selenium 在这类场景里通常更顺手

### 9.2 发布流程

标准流程：

```text
读取待发布记录
-> 启动持久化浏览器
-> 检查登录态
-> 打开小红书创作页
-> 选择图文发布
-> 上传图片
-> 填写标题
-> 填写正文
-> 补充话题
-> 检查页面状态
-> 点击发布
-> 记录结果和截图
```

### 9.3 Playwright 实现结构建议

模块建议：

- `sessionManager`
- `xhsPublisher`
- `uploadHelper`
- `publishValidator`
- `resultRecorder`

职责说明：

`sessionManager`

- 负责浏览器 profile
- 检查 cookie 是否有效
- 负责人工重新登录入口

`xhsPublisher`

- 执行实际发布流程

`uploadHelper`

- 处理图片上传
- 检查上传完成状态

`publishValidator`

- 检查标题字数、正文输入成功、图片是否全部加载

`resultRecorder`

- 保存截图
- 记录 note id 或发布结果

### 9.4 登录态管理

建议策略：

- 首次登录由人工扫码完成
- 保存浏览器 profile 到固定目录
- 每次发布前先检测是否仍处于已登录状态
- 如果登录失效，将任务标记为 `need_manual_login`

不要做：

- 每次任务重新登录
- 高频切换账号
- 多实例同时抢一个账号

### 9.5 选择器策略

建议：

- 优先用稳定文案和结构定位
- 不依赖随机 class 名
- 保留多套 fallback 选择器

例如：

- “发布笔记”
- “上传图文”
- “标题”
- “正文”
- “发布”

### 9.6 发布前校验

必须校验：

- 图片上传成功
- 标题不为空
- 正文不为空
- 页面未出现明显错误提示
- 当前账号状态正常

### 9.7 发布失败重试规则

建议：

- 同一条内容最多自动重试 `2` 次
- 每次失败后保存截图
- 如果是登录失效，直接停止自动重试
- 如果是上传失败，可重新打开页面重试
- 如果是页面结构变化，标记为系统错误待人工处理

### 9.8 账号风控控制

建议：

- 每天分多个时段发布，不要集中短时狂发
- 相邻两篇至少间隔 `20-40` 分钟
- 不要连续发布高度相似的内容
- 导流表达弱化，不要篇篇都出现强 CTA
- 预留人工暂停按钮

### 9.9 发布服务接口建议

## 11. 开发优先级与排期建议

### 第 1 周

- 建表
- 建立抓取配置
- 打通 2-3 个数据源抓取
- 做原始数据入库

### 第 2 周

- 做清洗聚类
- 做选题编排
- 接入文案生成
- 打通万相出图

### 第 3 周

- 做质检评分
- 做 Playwright 自动发布
- 打通截图、失败重试、结果回写

### 第 4 周

- 做调度优化
- 做风控节奏控制
- 做后台查看页
- 做小范围试运行

## 12. MVP 验收标准

达到以下标准即可认为第一版可上线：

- 至少接入 `3` 个外部热点来源
- 每天可自动生成不少于 `20` 个选题候选
- 每天可自动产出 `10-20` 篇图文草稿
- 每篇内容可自动生成至少 `1` 张封面图
- 自动发布成功率达到 `80%` 以上
- 所有失败任务都有截图和错误记录
- 可以通过开关暂停发布

## 13. 当前建议

从开发顺序上，不要先做复杂后台，也不要先做多平台抓取。最合理的顺序是：

1. 先打通 `抓取 -> 选题 -> 文案 -> 出图`
2. 再打通 `质检 -> 发布`
3. 最后补后台查看页和调度优化

这样可以最快验证整个链路，而不是把时间先消耗在外围工程上。
